title: MCP server
description: A built-in MCP server that exposes kftray port-forwarding operations to external clients.
---

kftray includes a built-in MCP server. When enabled, MCP-compatible clients can browse Kubernetes clusters, start and stop port-forwards, and manage saved configurations -- all through kftray.

## Enabling

<Steps>
<Step>
Open Settings in kftray.
</Step>
<Step>
Find the MCP Server section.
</Step>
<Step>
Toggle it on and set the port (default: 3000).
</Step>
</Steps>

The server runs in the background as part of kftray. It stops when kftray closes and auto-starts on next launch if left enabled.

## Connecting a client

Point an MCP client at:

```
http://localhost:3000/mcp
```

Clients that read server definitions from a config file typically need an entry like:

```json
{
  "mcpServers": {
    "kftray": {
      "url": "http://localhost:3000/mcp"
    }
  }
}
```

Where this file lives depends on the client. Check its documentation.

## What the server can do

16 tools, grouped in three areas:

**Cluster browsing** -- list available kubeconfig contexts, namespaces in a context, services in a namespace, pods in a namespace (with optional label filter), and ports exposed by a service.

**Port-forward control** -- start a forward (from a saved config or from scratch by specifying context, namespace, service, and port), stop a specific forward, stop all forwards, and list what is currently running.

**Configuration management** -- list, get, create, update, and delete saved configs. Export all configs as JSON or import configs from JSON.

All operations go through the same engine as the desktop and terminal interfaces. A config created through the MCP server shows up in the GUI and TUI, and a forward started through the MCP server is visible in those interfaces too.

<Callout type="info" title="Shared state">
All interfaces share the same database at `~/.kftray/kftray.db`.
</Callout>

## Examples

These show what a user might ask an MCP client and what kftray does behind the scenes.

### "What clusters do I have access to?"

The client calls `list_kube_contexts`. kftray reads the kubeconfig and returns the context names, for example: `dev-cluster`, `staging-cluster`, `prod-cluster`.

### "What services are running in the payments namespace on dev?"

The client calls `list_services` with the context and namespace. kftray queries the Kubernetes API and returns the service names, for example: `payments-api`, `payments-worker`, `payments-db`.

### "Connect me to payments-api on port 8080"

The client calls `start_port_forward` with the context, namespace, service name, and remote port. kftray creates a config, starts the forward, and returns the local port it picked (for example `localhost:54321`). If a specific local port was requested, it uses that instead.

If the forward fails to start, kftray deletes the config it just created so nothing is left behind.

### "Start config 3"

The client calls `start_port_forward` with just the config ID. kftray looks up the saved config and starts the forward. It returns the local and remote ports.

### "What's running right now?"

The client calls `list_active_port_forwards`. kftray returns a list of all active forwards with their config ID, service name, namespace, local port, remote port, protocol, and alias.

### "Stop the payments forward"

The client calls `stop_port_forward` with the config ID. kftray stops that specific forward and confirms with the service name and namespace.

### "Stop everything"

The client calls `stop_all_port_forwards`. kftray stops all active forwards and returns how many were stopped.

### "Save a config for grafana on port 3000 but don't start it yet"

The client calls `create_config` with the context, namespace, service, and ports. kftray saves the config and returns its ID. The forward is not started. It can be started later using that ID.

### "Forward to pods with label app=auth"

The client calls `start_port_forward` with `workload_type` set to `pod` and `target` set to `app=auth`. kftray finds a matching pod and starts the forward.

### "What ports does the redis service expose?"

The client calls `list_ports` with the context, namespace, and service name. kftray returns each port's name, number, target port, and protocol. If the service is not found, it falls back to looking at container ports on pods matching that name as a label selector.

## Tool reference

### Cluster browsing

| Tool | Required args | Returns |
|------|--------------|---------|
| `list_kube_contexts` | -- | list of context names |
| `list_namespaces` | `context` | list of namespace names |
| `list_services` | `context`, `namespace` | list of service names |
| `list_pods` | `context`, `namespace` | list of pods with name, status, and labels |
| `list_ports` | `context`, `namespace`, `service` | list of ports with name, port number, target port, and protocol |

Optional `kubeconfig` and `label_selector` (for `list_pods`) parameters are accepted on all tools.

### Port-forward control

| Tool | Required args | Returns |
|------|--------------|---------|
| `start_port_forward` | `config_id` **or** `namespace` + `remote_port` | success status, local port, remote port, config ID |
| `stop_port_forward` | `config_id` | success status, service name, namespace |
| `stop_all_port_forwards` | -- | success status, count of stopped forwards |
| `list_active_port_forwards` | -- | list of active forwards with config ID, service, namespace, ports, protocol, alias |

When creating a new forward (no `config_id`), these additional parameters are accepted: `context`, `service`, `target`, `local_port`, `protocol` (tcp/udp), `workload_type` (service/pod/proxy), `kubeconfig`, `alias`.

### Configuration management

| Tool | Required args | Returns |
|------|--------------|---------|
| `list_configs` | -- | list of configs with ID, alias, service, namespace, ports, protocol |
| `get_config` | `config_id` | full config details |
| `create_config` | `context`, `namespace`, `remote_port` | config ID |
| `update_config` | `config_id` | success status |
| `delete_config` | `config_id` | success status |
| `export_configs` | -- | all configs as JSON, count |
| `import_configs` | `configs_json` | success status |

## Security

<Callout type="warn" title="Local only by default">
The server binds to `127.0.0.1`. Only processes on the same machine can reach it.
</Callout>

The MCP server uses the same kubeconfig credentials as the rest of kftray. It does not store or transmit credentials separately.
